# SFSU Reservation System — Learning App Runbook (tiny single-table code app)

**Date:** June 18, 2026
**Purpose:** Make the abstract pieces (Dataverse table → generated typed service → delegation-safe
read → write-and-refresh) concrete by building the smallest possible end-to-end code app. Also a
real environment check: it proves the **write path** in a published code app, which the June 9
validation did not (that session only read `account`/`systemuser`).

**What it is:** a one-table app that lists the table's rows, has a text box + **Submit** button,
writes the typed text as a new row, then **re-reads the table** so the screen shows what Dataverse
actually stored.

**What it deliberately leaves out** (so you know these are real-project layers, not part of the
read/write loop): access gate, Dataverse security roles, the two group-access junctions, recurrence
/ conflict detection, centralized theming. None are needed to learn the core loop.

> ⚠ **Throwaway, not the real build.** Use a **disposable publisher prefix** (the default-solution
> auto-prefix, e.g. `crXXX_`). Do **NOT** spend `sfsures` here — that prefix is permanent per
> logical name and is reserved for the real tables (build sheet Step 0). You'll delete this table
> when done.

---

## Part A — Create the throwaway table (make.powerapps.com)

UI wording shifts between releases; the *goal* is fixed: create a table named **Message**, then
copy its logical name and its primary column's logical name.

1. Go to **make.powerapps.com**. Top-right **environment switcher** → select the **code-apps-enabled
   environment** (`orgdaa34530`). Targeting the wrong environment "works" but isn't where your app
   lives.
2. Left nav → **Tables** → **+ New table** → **New table**.
3. Name it **Message**. Save/create. Leave the **primary column** as the thing that holds the
   message text (its display name defaults to "Name"; that's fine — your typed message goes here).
4. **Copy two logical names** and write them down:
   - The **table** logical name — open the table → **Tools → Copy logical name** → e.g.
     `crXXX_message`.
   - The **primary column** logical name — in the column list → e.g. `crXXX_name`.
   > Notice the auto-prefix (`crXXX_`). That prefix is *permanent* on these names — exactly why
   > `sfsures` is a deliberate, one-time choice for the real tables. Here it's junk, so it doesn't
   > matter what it is, only that you copy it **exactly**.
5. (Optional but nice) Add a row or two by hand (**+ New row**) so the app shows something on first
   load — proves the *read* before you test the *write*.

## Part C — Connect the table

```bash
# Replace crXXX_message with YOUR table's actual logical name from Part A.
# Org URL: NO trailing slash.
npx power-apps add-data-source --api-id dataverse --resource-name crXXX_message --org-url https://orgdaa34530.crm.dynamics.com
```

Then **open `src/generated/services/` and `src/generated/models/`** and note the **exact filenames**
that were generated (singular table name in → pluralized files out, e.g. likely
`MessagesService.ts` / `MessagesModel.ts`). Use those exact names in your imports below. These files
are autogenerated — **don't edit them**. This is the per-table move you'll repeat for all ~13 real
tables; the generated model is your **compile-time schema guard** — it's what catches a column that
drifted from its intended type.

## Part D — The app (`src/App.tsx`)

Replace the contents of `src/App.tsx` with this. Then **find-replace** the placeholder
`learn_name` (appears 3×) with your **primary column's** real logical name (e.g. `crXXX_name`), and
fix the two import names if your generated files aren't `Messages...`.

```tsx
import { useEffect, useState } from 'react'
// ⚠ REPLACE these names with whatever actually appears in src/generated/ after add-data-source.
import { MessagesService } from './generated/services/MessagesService'
import type { Messages } from './generated/models/MessagesModel'
import './App.css'

function App() {
  const [rows, setRows] = useState<Messages[]>([])
  const [text, setText] = useState('')
  const [status, setStatus] = useState<'loading' | 'ready' | 'error'>('loading')
  const [saving, setSaving] = useState(false)
  const [errorMessage, setErrorMessage] = useState('')

  // READ the table. select + orderBy + top all DELEGATE to the Dataverse server.
  const loadRows = async () => {
    const result = await MessagesService.getAll({
      select: ['learn_name', 'createdon'], // ⚠ replace learn_name with your primary column's logical name
      orderBy: ['createdon desc'],          // newest first
      top: 50,
    })
    setRows(result.data ?? [])
  }

  // Initial load on mount.
  useEffect(() => {
    const init = async () => {
      try {
        await loadRows()
        setStatus('ready')
      } catch (err) {
        console.error('Load failed:', err)
        setErrorMessage(err instanceof Error ? err.message : String(err))
        setStatus('error')
      }
    }
    init()
  }, [])

  // WRITE one row, then RE-READ so the screen shows what Dataverse actually stored
  // (the "read the new data back from the table" refresh you asked for).
  const handleSubmit = async () => {
    const trimmed = text.trim()
    if (!trimmed) return
    setSaving(true)
    setErrorMessage('')
    try {
      // ⚠ The create() shape is the one new thing to confirm this session.
      //   Type "MessagesService.create({ " and let VS Code autocomplete list the
      //   valid field names — they come straight from the generated typed model.
      await MessagesService.create({ learn_name: trimmed }) // ⚠ replace learn_name
      setText('')          // clear the box
      await loadRows()     // re-read the table → UI reflects the source of truth
    } catch (err) {
      console.error('Save failed:', err)
      setErrorMessage(err instanceof Error ? err.message : String(err))
    } finally {
      setSaving(false)
    }
  }

  if (status === 'loading') return <p>Loading…</p>
  if (status === 'error') return <p>Couldn't load: {errorMessage}</p>

  return (
    <div style={{ maxWidth: 480, margin: '2rem auto', fontFamily: 'sans-serif' }}>
      <h1>Message board (learning app)</h1>

      <div style={{ display: 'flex', gap: 8, marginBottom: 16 }}>
        <input
          value={text}
          onChange={(e) => setText(e.target.value)}
          placeholder="Type a short message…"
          style={{ flex: 1, padding: 8 }}
        />
        <button onClick={handleSubmit} disabled={saving || !text.trim()}>
          {saving ? 'Saving…' : 'Submit'}
        </button>
      </div>

      {errorMessage && <p style={{ color: 'crimson' }}>{errorMessage}</p>}

      <ul>
        {rows.map((row, i) => (
          <li key={i}>{row.learn_name}</li> /* ⚠ replace learn_name */
        ))}
      </ul>
    </div>
  )
}

export default App
```

**If TypeScript complains about the `create({...})` argument:** that's the typed model doing its
job. Hover the red squiggle, or open `src/generated/services/MessagesService.ts` and read the
`create` method's parameter type — it tells you the exact input shape. Adjust the field name to
match. (This is the same "let the compiler enforce the schema" behavior you'll lean on for the real
tables.)

**Do NOT add** `import { initialize } from '@microsoft/power-apps/app'`. This template
auto-initializes the SDK at startup and that member isn't exported in the pinned version — adding it
throws. There is no `PowerProvider.tsx` in this template; that's expected.

## Part E — Run, then publish, then test the write path

```bash
# Run locally. Open the "Local Play" URL (NOT the plain localhost Vite link),
# in the same browser profile signed into the tenant. Allow the localhost
# permission if prompted (Chrome/Edge block local network access by default since Dec 2025;
# suspect this first if it hangs on a spinner).

```

Test locally: you should see the existing rows; type a message; click **Submit**; the new row
appears in the list. Then publish:

```bash
npm run build           # clean green build = a free correctness check
npx power-apps push     # returns the live play URL
```

Open the **published** play URL and repeat type → Submit → appears. **This published round-trip is
the real verification** — a code app *writing* to Dataverse and reading it back, live in the tenant.
(Preview note: a published code app can intermittently fail one refresh and work the next — refresh
a few times before worrying.)

## Part F — What this made concrete (the mapping to the real build)

- **`add-data-source` → generated typed service/model:** exactly how each of the ~13 real tables
  gets wired. The generated model is your compile-time schema guard.
- **`getAll({ select, orderBy, top })`:** the delegation-safe read shape that conflict detection and
  reports *must* use so they run server-side instead of pulling a capped local batch.
- **`create` + re-read:** the write path (previously unproven in a code app) and the honest
  "refresh from the source of truth" pattern.
- **The auto-prefix you saw on the logical names:** why `sfsures` is a deliberate, permanent,
  one-time choice for the real tables (build sheet Step 0).

What you did **not** touch here, and why it's fine: no security roles (the *real* data boundary —
this throwaway just rides your own System Admin access), no access gate, no junctions / recurrence /
conflict detection, no theming. Those are the production layers; this app exists to make the
plumbing tangible, not to model the domain.

- **New terminal** after installing Node/git (PATH).
- At `init`, sign in with the **Microsoft / SFSU** account, **not** GitHub.

- **Allow localhost** if prompted (Chrome/Edge block it by default since Dec 2025).
- Org URL **no trailing slash**. `fetch failed` is also a known intermittent — retry once or twice;
  add `--json` to a CLI command for a rawer error.
- `npm warn` (yellow) = noise; only `npm error` (red) matters. Don't `npm audit fix` the preview
  template.
- Don't add `import { initialize }`; there's no `PowerProvider.tsx` — both expected.
